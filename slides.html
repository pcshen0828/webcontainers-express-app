The Problem with Traditional Approaches 傳統方案的痛點

Cloud VM sandboxes work, but come with serious trade-offs. Cloud VM 沙盒雖然可行，但帶來了不少代價。

Traditional Cloud VM傳統 Cloud VM
  • Network round-trips on every action — noticeable latency每個操作都需網路往返，延遲明顯
  • Expensive server infrastructure that must scale with users需要昂貴的伺服器基礎設施，隨使用者數量增加
  • User code runs on shared servers — security risk使用者程式碼跑在共享伺服器上，存在安全風險
  • Slow boot time — spinning up VMs takes seconds啟動緩慢，開啟 VM 需要數秒
WebContainers
  • Zero latency — everything runs locally in the browser tab零延遲——所有運算直接在瀏覽器分頁內完成
  • No server cost — uses local compute, distributes via CDN無伺服器費用——使用本機運算，透過 CDN 分發
  • Isolated by design — runtime contained within the browser sandbox天然隔離——執行環境封閉在瀏覽器沙盒內
  • Boots in milliseconds — no VM provisioning needed毫秒級啟動——無需等待 VM 佈建

Proven at scale — powering millions of StackBlitz users and the interactive docs of Svelte, Angular, and Nuxt. 已通過大規模驗證——驅動數百萬 StackBlitz 使用者，並支援 Svelte、Angular、Nuxt 的互動式官方文件。

What is
WebContainers? 什麼是
WebContainers？

A browser-native runtime that executes Node.js, npm, and shell commands entirely inside the browser tab. Built by StackBlitz, it powers StackBlitz.com. 一個瀏覽器原生的執行環境，可直接在瀏覽器分頁內執行 Node.js、npm 與 shell 指令，無需任何後端。由 StackBlitz 開發，為 StackBlitz.com 提供技術核心。

Under the Hood 底層架構

WebContainers stack four modern browser primitives. WebContainers 建立在四項現代瀏覽器底層技術之上。

Layer 1

WebAssembly (WASM)

The Node.js runtime binary (C/C++) can't run in browsers directly — browsers only execute JavaScript and WASM. So the C/C++ parts are compiled to WASM; Node.js's built-in JS modules still run in the browser's own V8, avoiding double-VM overhead. OS-level calls like open and fork are intercepted and re-implemented with browser APIs.Node.js 的 runtime binary 以 C/C++ 撰寫，瀏覽器無法直接執行——瀏覽器只能跑 JavaScript 與 WASM。因此 C/C++ 部分編譯為 WASM，Node.js 的 JavaScript 內建模組仍由瀏覽器原生 V8 執行，無雙重虛擬機損耗。open、fork 等 OS 呼叫則被攔截並以瀏覽器 API 重新實作。

Layer 2

SharedArrayBuffer & Atomics

SharedArrayBuffer is a block of memory shared across Workers — any Worker can read or write it directly, with no message-passing overhead. Atomics provides synchronization on top: Atomics.wait() blocks a Worker until notified, and Atomics.notify() wakes it up. Together they let WebContainers emulate synchronous blocking calls (e.g. readFileSync) across Worker boundaries.SharedArrayBuffer 是可跨 Worker 共享的記憶體區塊——任何 Worker 都能直接讀寫，無需訊息傳遞。Atomics 在此之上提供同步機制：Atomics.wait() 讓 Worker 阻塞等待，Atomics.notify() 將其喚醒。兩者合力讓 WebContainers 能在跨 Worker 的環境中模擬 Node.js 同步阻塞呼叫（如 readFileSync）。

Layer 3

Service Worker

The container's HTTP server has no real TCP port — it runs inside a WASM Worker. When a port opens, WebContainers generates a unique HTTPS proxy URL (e.g. *.webcontainer-api.io) for it. The Service Worker intercepts fetch requests to that URL, forwards them to the WASM Worker, and returns the response — so the <iframe> behaves as if it's talking to a real server.容器內的 HTTP 伺服器沒有真實 TCP 連接埠，跑在 WASM Worker 裡。當 port 開啟時，WebContainers 會為它產生一個專屬的 HTTPS 代理網址（如 *.webcontainer-api.io）。Service Worker 攔截送往該網址的 fetch 請求，轉發給 WASM Worker 並回傳回應——<iframe> 的行為就像在存取一個真實的伺服器。

Layer 4

Web Workers

Browsers run JS on a single main thread — any blocking operation freezes the UI. WebContainers gives each spawned process its own Web Worker, enabling true parallelism (e.g. running a dev server and npm install simultaneously). The WASM runtime must also run in a Worker because Atomics.wait() — required by Layer 2 — is forbidden on the main thread.瀏覽器的 JS 預設只有一條主執行緒，任何阻塞操作都會導致畫面卡住。WebContainers 為每個子 Process 配置獨立的 Web Worker，實現真正的平行執行（例如同時跑 dev server 與 npm install）。WASM runtime 也必須跑在 Worker 裡，因為 Layer 2 依賴的 Atomics.wait() 在主執行緒中是被禁用的。

Cross-Origin Isolation 跨來源隔離

After Spectre/Meltdown attacks (2018), browsers disabled SharedArrayBuffer by default as a side-channel mitigation. Cross-Origin-Opener-Policy (COOP) + Cross-Origin-Embedder-Policy (COEP) signal that the page is cross-origin isolated — the browser then re-enables SharedArrayBuffer, which WebContainers depends on for synchronous Node.js APIs. Without these headers, WebContainer.boot() throws immediately. 2018 年 Spectre/Meltdown 攻擊事件後，瀏覽器預設停用 SharedArrayBuffer 以防範旁路攻擊（side-channel attack）。

設定 Cross-Origin-Opener-Policy（COOP）+ Cross-Origin-Embedder-Policy（COEP）標頭可向瀏覽器宣告頁面已完成跨來源隔離，瀏覽器才會重新啟用 SharedArrayBuffer——這正是 WebContainers 實作 Node.js 同步 API 的基礎。

缺少這兩個標頭，WebContainer.boot() 將直接拋出例外。

Cross-Origin-Opener-Policy: same-origin
Cross-Origin-Embedder-Policy: require-corp
⚠ COEP blocks cross-origin subresources (images, fonts, scripts…) unless they respond with Cross-Origin-Resource-Policy: cross-origin. CDN assets without this header will fail to load.COEP 會封鎖所有跨來源子資源（圖片、字型、script…），除非該資源回應中包含 Cross-Origin-Resource-Policy: cross-origin。 未加此標頭的 CDN 資源將無法載入。
